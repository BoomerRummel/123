# Rummel的新手任务经验总结

**写在开头：**

如果现在阅读文档的你是一名完全不懂编程的医学专业学弟/学妹，我希望你不需要因代码能力而感到慌张，因为AI的辅助编程能力能够完全弥补这一点（本人非CS专业，代码能力也较为薄弱）；
而我认为更需要下功夫的是**提示词的准确性**，包括但不限于角色定位，背景普及，任务要求等等，所有提示词都对AI的回答至关重要，耐心与AI对话，不急于求成，最后的结果可能才会比较符合预期。

## 工具篇

### **AI选择：**
各AI的优劣势在前辈的**新手入门**文档中已有较为全面的总结，现就我个人的使用经验对主流三款大模型进行简单分析。

1.**Chatgpt:**
长文本和复杂任务能力较强，而且自从Image-2的更新后它基本就是全能战士了，弥补了先前生图质量不如Gemini Nano Banana的缺点，在文本生成，代码编写，图像生成等各方面能力全面，十分推荐使用。
缺点可能就是账号略贵，单账号20美刀一个月。

2.**Gemini:**
Gemini也是我本次任务的主要科研助手，但是用下来给我的感觉是虽然近期界面交互优化了，并且卡顿现象减少了，但是似乎模型的注意力没有以前好了，回答质量有所下降（朋友使用体验也是一致）。而且似乎Nano banana2 的生图质量也不及我预期（最后的图像生成还是用GPT做了）。

虽然但是，它的优点在于谷歌的**AI生态联动**做的很优秀，NotebookLM和Gemini的联动能够为日常学习复盘，以及信息分析整理等任务带来很大的生产力, 这是谷歌独特的一点。

3.**Claude:**
CC是当今大模型的领头羊，模型能力是公认的第一，同时相较于上述两个LLM：它能够本地化部署并作为agent自动化针对电脑文件进行任务交付是其他模型所没有的能力（自主化程度高）。

作为当今最强大的生产力工具，它的缺点也显而易见：**门槛高**。
这里的高门槛是各个方面，不同层次的：
1.**本地化部署**：对于小白来说有一定难度，包括环境配置，API接入等前期工作。
2.**账号获取难度高**：claude账号不支持国区手机号码注册，而且账号风控很严格，中国用户使用很容易被封。
3.**模型开销大**：强大的模型能力背后是更大的开销；完整版的CC用的原生账号tokens计费很昂贵，相较之下接入国产大模型（千问，智谱等）不失为一种更具性价比的选择，但是这又需要解决兼容度的问题（安装CC switch或其他中转站）。
4.**使用者能力门槛高**：强大的模型对使用者的要求也更高。首先claude.md的质量就决定了整体生产的质量，其次cc支持使用者在工作中继续跟cc进行交流，这就需要使用者对整个工作流程更清楚。

### **提示词技巧：**

1.**前置工作**：
对于每次任务我都习惯先让AI明确角色定位，背景，任务等信息，参考模板：

```
    1.角色设定：...
    2.背景设定：...
    3.任务设定：...
    4.特别要求：所有回答和答案在回复前先自我检查，保证准确无误，有所根据，避免虚假答案的出现。
    在没有必要的情况下不过度回答，精确无误地回答我的问题即可。
    以上为这次工作的所有background设定，了解后等我告诉你我准备好了再开始协助工作。 
```

2.**辅助编程**：

* 辅助编程的关键就是**拆分任务**，将一个大任务拆分成若干个小任务让AI依次编程，以集中注意力高效率完成每个小环节，最后集成结果会比直接把一个大的编程任务让AI处理好很多；因为不仅是AI注意力集中，在每个小任务的交付后你都可以适当给予AI反馈，相当于一个微调过程，能够更精确地引导AI产出符合预期的结果。

* 其次，让AI在每个代码后加上注释，不仅是帮助自己理解，也是帮助他人理解代码逻辑。（对于代码基础薄弱的同学，每次AI辅助编程也都是一次学习的机会，尽力做到**不**不求甚解）

示例：

```
    # 1. 加载数据
    data_path = Path("heart_failure_clinical_records_dataset.csv")
    df = pd.read_csv(data_path)

    # 2. 准备特征与目标
    # 将 DEATH_EVENT 作为目标变量
    X = df.drop('DEATH_EVENT', axis=1)
    y = df['DEATH_EVENT']

    # 3. 构建并训练随机森林模型
    # 设置 random_state 保证可重复性，这对学术论文至关重要
    rf_model = RandomForestClassifier(n_estimators=100, random_state=42, max_depth=5)
    rf_model.fit(X, y)
```


## 实战篇

1.**Debug:**

* 当遇到跑不通的代码，可以不急着把报错代码直接喂给AI；可以先跟AI重述一下任务目标，或者背景，再说明报错信息。让AI回顾一下上下文信息，再针对代码进行debug，否则很有可能出现debug后其他代码消失/功能受损等问题。
X

* 如Debug中所述，**定期让AI回忆上下文**是很有必要的，能够很大集中AI注意力，降低幻觉，从而提高回复质量，提升工作效率（当任务不断推进，噪音不可避免地会累加，稀释AI注意力）。

我的建议是每完成2-3个小任务后用以下prompt进行任务回溯：

```
    "为了确保我们的协同高效，请对我们刚才的对话进行一次任务复盘（Retrospective）。请按以下结构输出：
    1.当前目标： 我们最终要实现的业务/技术目标是什么？
    2.已解决事项： 我们已经处理完的问题或达成的共识。
    3.遗留问题/待办事项： 哪些事情还没做？哪些报错还没完全排查完？
    4.下步行动建议： 基于现状，你认为我接下来应该优先处理哪一件事，以及为什么？"
```

3.**多开窗口，多线程工作：**

* 在处理不同方向任务可以选择多开窗口/AI模型，多线程处理（本质上也是最大化利用模型注意力，减少噪音）。
但是需要注意的是，需要同一上下文的背景信息任务自然应该在同一窗口执行，若想要多开窗口应同步上下文信息。


## 进阶篇

1.**AI生图：**

* 模型选择方面我现在更推荐GPT的Image-2，在使用Nano banana 2的过程中我感觉模型对指令的理解有所欠缺，不能很好的对科研流程图进行修改，甚至还会出现幻觉：包括但不限于虚假信息，中英文夹杂，拼写错误等问题（可能和模型注意力涣散有关）。

但是gpt的Image-2效率就会高效很多，同时能够给出多种版本供我选择，风格也更符合科研taste.

2.**生图幻觉解决：**

* 幻觉解决问题个人还没有找到很有效的解决方法，一方面可能是尽可能选择能力较强的生图模型，一方面是尽可能优化自己的提示词，给AI足够的上下文信息，对AI的回答进行足够的限制。

* 其次，如果能够自己手作一张基础流程图，喂给AI让他在基础上优化也能够一定程度提高效率。
